Security Model
Three independent security layers — network, authentication, and data. A breach of one layer does not compromise the others.
Layer 1 — Network
Cloudflare HMAC Validation
Every Asana webhook request is validated before reaching Supabase. Forged or replayed requests are rejected at the Cloudflare edge.
MethodHMAC-SHA256 signature on request body
Secret per projectStored in asana_webhook_config.hook_secret
Failure responseHTTP 401 — request dropped
CoverageAll 15 registered Asana webhooks
Layer 2 — Authentication
Supabase Auth + Domain Routing
Email-based auth with automatic role assignment. No manual role configuration — domain suffix determines access tier.
Admin access@rampfuel.com — full cross-client access
Client accessAll other emails — scoped to their client only
Client linkingclient_users table (auth_user_id → client_id)
Admin usersdavid, jared, cole, cami @rampfuel.com
Layer 3 — Data
Row Level Security (RLS)
Enforced at the PostgreSQL layer — not the application layer. Even if portal code had a bug, cross-client data leaks are impossible.

Key Policies

tasks — Client read policy
EXISTS (
  SELECT 1 FROM projects p
  JOIN client_users cu ON cu.client_id = p.client_id
  WHERE p.id = tasks.project_id
  AND cu.auth_user_id = auth.uid()
)
All tables — RampFuel admin policy
auth.email() ILIKE '%@rampfuel.com'
Key guarantee
A Lexful user querying the tasks table will only ever receive rows where projects.client_id = Lexful — even via direct API calls. The database enforces this, not the portal.

v1.2 — Upcoming
Gwen: Meeting Processing Layer
Gwen enters the pipeline between Fellow.app and the tasks table — ingesting meeting transcripts, extracting structured work, deduplicating against existing tasks, and routing candidates to a human approval queue before anything goes live.

What changes from current architecture

BEFORE (current)
Fellow.app writes action items directly to Asana as task notes. Asana webhooks sync them to Supabase. No processing, no deduplication, no intelligence — the full volume lands in the DB and portal as-is. Jared and David interpret everything manually.
AFTER (v1.2)
Gwen intercepts the meeting output, runs semantic dedup against existing open tasks, and stages candidates for human review. Nothing writes to production tasks without an approver signing off. Jared and David review decisions — they don't make them from scratch.

When Gwen detects a task that requires a deliverable, it triggers a Claude generation pipeline — drafting content against the client's full engagement history.

Design Principles

Supabase is the source of truth
Asana is the operational sync layer. Every design decision assumes Supabase owns the data.
Human approval at every AI boundary
Gwen never writes to production tables without a human sign-off. No exceptions in v1.2 or v1.3.
Additive migration only
New tables extend the existing schema. The current portal keeps working through every version transition.
RLS enforced at every layer
Every new table in v1.2 and v1.3 gets RLS from day one. Client isolation is non-negotiable.

Migration Strategy

PhaseWhat happensPortal impact
Phase 1 PMS tables added alongside Asana sync — dual-write None — existing queries unchanged
Phase 2 Portal queries switch from Asana-synced tasks to PMS tasks Zero downtime cutover
Phase 3 Asana becomes optional — webhooks write to PMS, not tasks directly Richer data, same portal interface
Phase 4 Asana removed from critical path PMS is now the complete operational system
